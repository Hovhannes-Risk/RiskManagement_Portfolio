# Data Dictionary

*Definitions for every field used across the portfolio. Includes raw
bet-level fields, derived analytical fields, and risk-scoring outputs.*

---

## 1. Raw bet-level data (`mydata.xlsx.xlsx`, sheet `Raw_Data`)

| Field | Type | Description | Notes |
|---|---|---|---|
| `bet_id` | string | Unique identifier for each bet ticket | Primary key |
| `bettor` | string | Synthetic account ID (`P00001`…`P12208`) | Generated — no real identity exists behind any ID |
| `bet_time` | datetime | Timestamp of bet placement | Parsed to `bet_datetime` downstream |
| `sports` | string | Sport category | e.g. Football, Tennis, Basketball, Counter-Strike 2 |
| `league` | string | Competition / league | e.g. Premier League, Champions League |
| `market` | string | Market type | e.g. "Full time result", "Over/Under", "Both teams to score" |
| `usd_amount` | float | Stake in USD | The amount the player risked |
| `usd_ggr` | float | Gross Gaming Revenue (USD) | **Positive = house won; negative = house lost.** Per-bet P&L for the operator |
| `odds` | float | Decimal odds at placement | Standard decimal format (e.g. 2.50 for +150 American) |
| `resolved_odds` | float | **Net return multiplier** | See critical note below |
| `subbet_result` | string | Settlement state | `won` / `lost` / `cancelled` / `cashout` |

### Critical note — `resolved_odds`

This field is **NOT** the decimal odds. It is the net return multiplier
(profit per unit stake):

- `won` → `resolved_odds = decimal_odds − 1`  (e.g. a 2.71-odds win stores 1.71)
- `lost` → `resolved_odds = 0.00`
- `cancelled` → varies by operator convention
- `cashout` → partial multiplier reflecting the agreed payout

To compute average decimal odds, use `mean(resolved_odds + 1)` over
won bets only. See `methodology.md` for the full derivation.

### Critical note — `usd_ggr` sign convention

This portfolio uses the **operator-side** convention:

| Outcome | `usd_ggr` sign |
|---|---|
| Player loses (house keeps the stake) | **Positive** |
| Player wins (house pays out) | **Negative** |
| Push / void | Zero |

A negative aggregate GGR therefore means the **house** is losing money,
not the players. This is opposite to the player-side P&L convention
used in some retail-facing reports.

## 2. Data provenance

There is no anonymization mapping, because there is nothing to
anonymize: every account is synthetic, generated by
`data_generator/generate_sportsbook_data.py` (seed 42). The generator
plants the documented risk patterns (same-second bots, low-volume
sharps, a thin-league insider group, coordinated pairs) inside a
recreational mass; planted segments occupy the high ID range
(`P11554`+), which the test suite uses as ground truth for recall
checks.

## 3. Player risk scoring output (`player_risk_scores.xlsx`, sheet `All_Players`)

| Field | Type | Description |
|---|---|---|
| `Rank` | int | Position in the sorted risk table (1 = highest risk) |
| `Bets` | int | Number of settled bets in observation window |
| `Total_Stake` | float | Sum of `usd_amount` across all bets |
| `GGR` | float | Sum of `usd_ggr` across all bets (negative = player won net) |
| `Avg_CLV` | float | Closing-line-value proxy (0 if not available) |
| `Win_Rate` | float | `won_bets / settled_bets`, range 0–1 |
| `Risk_Score` | float | Composite normalized score, range 0–1 |
| `Risk_Level` | string | Band derived from `Risk_Score` |

### Risk_Level bands

| `Risk_Score` range | `Risk_Level` |
|---|---|
| ≥ 0.70 | CRITICAL |
| 0.50 - 0.69 | HIGH |
| 0.30 - 0.49 | MEDIUM |
| 0.15 - 0.29 | LOW |
| < 0.15 | MINIMAL |

## 6. Master analysis output (`master_analysis.xlsx`)

Multi-sheet workbook with the following sheets:

| Sheet | Contents |
|---|---|
| `Client_KPIs` | Per-player KPIs including `Avg_Won_Odds` |
| `Sport` | GGR, hold %, bet count by sport |
| `Market` | Same breakdown by market type |
| `League` | Same breakdown by league |

Column conventions match the raw data; `Avg_Won_Odds` is computed
per the methodology document (won bets only, `+1` adjustment).

## 7. Dashboard / Report derived metrics

| Metric | Formula | Range |
|---|---|---|
| `Hold %` | `100 × GGR / Total_Stake` | Typically -20% to +15% |
| `Bot Suspects` | Distinct bettors with ≥2 bets in the same second | 0 to total players |
| `Hold` (alternative) | Same as Hold % expressed as decimal | -0.20 to 0.15 |

A negative Hold % means the house is losing money on the segment. A
healthy long-run Hold % for a balanced sportsbook typically sits in the
3-7% range; sustained values below this indicate either pricing issues,
sharp-account dominance, or both.

---

## 8. Conventions

- **Currency.** All monetary values in USD unless otherwise noted.
- **Time.** All timestamps assumed to be in a single zone; DST
  boundaries not separately handled.
- **Missing values.** Left blank rather than zero-filled in fields
  where zero has a different meaning (e.g. average odds for
  never-winning players).
- **Rounding.** Display rounded to 2 decimal places; underlying values
  retain full precision.
